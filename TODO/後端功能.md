## G1. 即時戰況推播 — ✅ 已做（payload 精簡）

`GET /api/state/ws` 現在同一條連線推**兩種幀**，以 `type` 欄位區分：
- **狀態幀**：`{ "state": "...", "updated_at": "...", "end_time"?, "scores"? }`（無 `type`）。
- **slot_update 幀**：`{ "type": "slot_update", "user_id": 2 }` — 只說「該使用者的 slot 有變動」（移動／攻擊／修復都會推），廣播給所有訂閱者。

**前端對齊**：
- `ApiFightService.watchEvents()` 連 `/api/state/ws`、只取 `slot_update` → 收到就 refetch 全體狀態更新地圖。
- ⚠️ **修了一個 bug**：`ApiGameStateService` 原本會把 `slot_update` 幀當成缺 `state` → 誤判成 `NORMAL`，害遊戲階段亂跳。已改為「非狀態幀（有 `type` 或無 `state`）一律略過」。
- 目前 GameState 與 Fight 各開一條 WS 到同一端點（皆可用；未來可合併成一條共用以省連線）。

**仍缺（影響需求 4 / 注意事項 1）**：slot_update **不含攻擊者是誰、打了哪一格、哪個元件**。
- 注意事項 1（正在打的格被搶先打掉要跳掉）：可行 — 收到 `slot_update(該敵組)` 後 refetch，發現我正在打的 (user, slot) 已 broken 就跳掉。
- 需求 4（App 內通知「哪組攻破了我們的什麼元件」）：**只能**由前端比對自己組前後快照推出「我方某元件被攻破」，**無法顯示是哪一組**。
  → 若要顯示攻擊方，需後端在 `slot_update` 加 `actor_user_id`（+ 可選 `slot_id` / `item_type`）。**待確認是否補。**

## G2. 全體組別狀態彙整端點 — ⬜ 未做

後端未提供彙整端點。前端 `ApiFightService.fetchAllGroups()` 以 `GET /api/users`（取 role=0）+ 對每組 `POST /api/item` 拼出（N+1、非原子）。功能可用；組數不多時可接受。若日後要省請求／要原子快照可再評估補。

## G3. 血量定義 + 排行榜端點 — ✅ 已做

- **血量定義（實作）**：`score` = 該組**未損毀** slot 內元件所綁**題目的 difficulty 加總**（broken 算 0）。即「剩餘血量」。
- **端點 `GET /api/scores`** → `{ "scores": [ { "user_id", "score" } ] }`。**QUIZ2 結束時重算一次**（顯式切換或排程 auto-revert），非每次即時。每位使用者都有一列（無分數記 0）；首次 QUIZ2 結束前回空陣列。
- 狀態幀 / `GET /api/state` 也會夾帶最近一次的 `scores`。

**前端對齊**：
- `fetchLeaderboard()` 改打 `GET /api/scores`，再合併 `GET /api/users` 補 display_name / 頭像、名次前端排（需求 3 僅需名次/頭像/組名/剩餘血量；端點未給「上限」，排行榜只顯示剩餘血量）。
- 空陣列（首場未結束）→ 後備用目前全體狀態自算。
- 注意：**世界地圖即時血量條**是前端用「元件設定難度」即時估算（因 `scores` 只在 QUIZ2 結束才更新）；最終排行榜以 `GET /api/scores` 為準。兩者在「元件設定難度 == 題目難度」時一致。

## G4. 每格禁打記錄 — ✅ 已做

- `POST /api/item` 每個 slot 多回 `blocked_attackers`：被禁止攻擊這格的 user_id 清單（**空時省略**，全視圖與受限視圖都回）。
- 失敗攻擊 → 加入該 slot 的 `blocked_attackers`；該攻擊方在**修復前**不能再 `target` 這格（`POST /api/question/target` 回 403）。
- 修復該格 → 清空整個 `blocked_attackers`；切回 `NORMAL` → 清空所有格的禁打。

**前端對齊**：`ApiFightService._fetchSlots()` 讀 `blocked_attackers`，對「我」而言 `attackBlocked = 清單含 selfUserId`。可攻擊判定 = `!broken && !attackBlocked`。
（攻擊流程下一階段做：`targetQuestion` 收到 403 要提示「這格你已答錯，需等對方修復」。）

---

## G5. 修復換題 + 難度對應 — ✅ 已做

修復（`target` 打自己壞掉的 slot）現在會抽「**該損毀元件難度**的 area2 題」（該難度的 area2
題須存在，否則 400）；**答對後把該題綁回 item**——修好的部件之後若被攻擊，即以這題計分 / 出題。

**前端對齊**：補給站修復走 `QuizService.targetQuestion(targetUserId=自己, slotId)` → 作答 →
`QuizResult.success`。前端不需感知難度過濾（後端依壞掉元件自動決定）；修復成功後 refetch 反映。

---

## G6. 攻防倒數結束時間 — ✅ 已具備

`POST /api/state` 支援 `end_time`，狀態幀會帶。前端已解析成倒數並在時間到觸發結算。**老師切 QUIZ2 務必帶 `end_time`**，否則前端只能用後備固定長度、且不會自動結束。

## 沿用的既有端點（已確認對齊）

- `POST /api/question/target`（攻擊／修復開 session；攻擊別組未損毀格、修復自己壞格）。前端 `QuizService.targetQuestion()`。
- `POST /api/question/answer`（回 `correct`；target session 另回 `success` = break/repair 是否成立；答錯攻擊會被加入該格 `blocked_attackers`）。前端 `QuizResult.success` 已接。
- `POST /api/item`（自己全視圖含 broken / 別組受限視圖）。
- `GET /api/users`（role=0 組的 display_name / 頭像 / building）。
