## G1. 即時「戰況」推播（WebSocket）　⬜

**問題**：現有 `GET /api/state/ws` 只推遊戲階段（NORMAL/QUIZ1/QUIZ2），不推任何「誰打了誰、哪格壞了／修了」。
需求 2/3/4 與注意事項 1、5 全靠「有人被打或修補時即時通知，前端再去 refetch 全體狀態」。

**需求**：攻防戰期間，任一組 slot 發生 break / repair 時，向所有訂閱者推一則「戰況事件」。
可擴充現有 `/api/state/ws`（多一種訊息型別），或開新 WS — 由後端決定，前端兩種都能配合。

**建議事件格式**（與現有 `{state, updated_at}` 並存，前端用是否有 `event` 欄位區分）：

```json
{
  "event": "attack" | "repair",
  "actor_user_id": 5,
  "actor_display_name": "胖仔大砲蛙隊",
  "target_user_id": 2,
  "target_slot_id": 3,
  "item_type": 10,
  "broken": true
}
```

**前端用途**：
1. 收到事件 → 去抓全體狀態（G2 或 N×`POST /api/item`）即時更新世界地圖。
2. `target_user_id == 自己` 時，顯示 App 內通知「`actor_display_name` 攻破了我們的『元件名(item_type)』」（需求 4）。
3. 自己正在打的 `(target_user_id, target_slot_id)` 被別人先打掉（收到他人 `attack` 事件）→ 把作答畫面跳掉（注意事項 1）。

## G2. 一次取得「所有組別狀態」的彙整端點　⬜（可選）

**問題**：現在要拼世界地圖只能 `GET /api/users`（取所有 role=0）再對每組各打一次 `POST /api/item`（N+1 次、非原子）。

**建議端點**（有最好；沒有的話前端 fallback 用 N×`POST /api/item`）：

```
GET /api/fight/state
```

```json
{
  "groups": [
    {
      "user_id": 2,
      "display_name": "野仔大樹蛙隊",
      "profile_pic_url": "/images/abc.jpg",
      "building_id": 1,
      "hp": 9,
      "hp_max": 13,
      "slots": {
        "0": { "type": 10, "broken": false, "attack_blocked": false },
        "3": { "type": 22, "broken": true,  "attack_blocked": false }
      }
    }
  ]
}
```

- `slots` 對所有人公開「type + broken」即可（不需 item_id / question_id）。
- `attack_blocked`：**對呼叫者（caller）這一組而言**，這格是否因先前答錯而禁止再攻擊（見 G4）。
- `hp` / `hp_max`：見 G3 定義（後端算好直接給，前端也可自算，但以後端為準）。

## G3. 血量定義 + 排行榜／結算端點　🔧

**血量定義（已確認）**：
- **上限 `hp_max`** = 該組**所有已放置物件**的難度分數（1/2/3）加總。
- **剩餘 `hp`** = 其中**未損毀**物件的難度分數加總。
- 物件難度 = 該 `type` 在所屬 building `difficulty_type` 內的難度層級。

**排行榜／結算端點**（後端會提供，請定義回傳）：時間到後前端呼叫，顯示 名次／頭像／組名／剩餘血量。建議：

```
GET /api/fight/leaderboard
```

```json
{
  "results": [
    { "rank": 1, "user_id": 5, "display_name": "胖仔大砲蛙隊", "profile_pic_url": "/images/x.jpg", "hp": 13, "hp_max": 13 },
    { "rank": 2, "user_id": 2, "display_name": "野仔大樹蛙隊", "profile_pic_url": "/images/y.jpg", "hp": 9,  "hp_max": 13 }
  ]
}
```

- 排序與名次由後端決定（同分如何排序請後端定義，例如同分並列或以 user_id 次序）。

## G4. 每格「禁打」記錄（答錯鎖 + 可攻擊提示）　🔧

**規則（注意事項 3）**：某組對某 slot 的「目前這題」答錯後，在該格題目被換掉（被修復換題，見 G5）之前，這組不能再打這格。

**後端已著手**：在 slot 資訊記錄「哪些組不能打這格」。

**前端需要的暴露方式**：在「查別組 slot」的回應（`POST /api/item` 受限視圖，或 G2 的 `/api/fight/state`）每格加一個布林：

```json
"attack_blocked": true   // 對呼叫者這一組而言，這格目前禁打
```

> 只需回「caller 自己是否被擋」即可，不必把所有被擋的組 id 列給學生。
> 前端用它同時做兩件事：(a) 注意事項 3 的禁打；(b) 需求 2「提示可攻擊部位」— `broken == false && attack_blocked == false` 才標記為可攻擊。

## G5. 修復換題 + 難度對應　⬜（後端之後會加）

**現況**：`POST /api/question/target` 打自己壞掉的 slot 走修復，回「隨機 area2 題」，未依難度過濾，也未把 item 綁定題目換新。

**需求（需求 22–23、注意事項 2）**：
1. 修復取題依**壞掉元件的難度**過濾（不是隨機 area2）。
2. 修復**答對**後，把該 item 綁定的 `question_id` 換成這次答對的題目 — 之後別人來打這格就打這題。

> 連帶效果：item 換題 = G4 的「題目版本」改變 → 該格對所有先前答錯者解除禁打。

## G6. 攻防倒數結束時間　✅（已具備，需老師端帶值）

`POST /api/state` 已支援 `end_time`，`GET /api/state` 與 `/api/state/ws` 會回 `end_time`，前端 `ApiGameStateService` 已解析為倒數。

**檢查項**：老師端切 QUIZ2 時**務必帶 `end_time`**，否則前端只能用後備固定長度，且時間到不會自動回 NORMAL。

## 前端會用到的既有端點（確認沿用，無需改動）

- `POST /api/question/target`（QUIZ2 開攻擊／修復 session）— 攻擊：別組未損毀 slot；修復：自己壞掉 slot。
- `POST /api/question/answer`（送作答）— 回 `correct`，target session 另回 `success`（break/repair 是否成立）。前端會讀 `success` 判定攻擊／修復成功與否。
- `POST /api/item`（自己組全視圖：含 broken）— 自己島嶼「剩餘／損毀元件清單」、補給站「已損元件清單」。
- `GET /api/users`（所有 role=0 組：display_name / profile_pic_url / building_id）— Loading 頁頭像、地圖氣泡。
- building `difficulty_type` / `item_allowed_slot` / layout（已由 `GET /api/building/{id}` 取得）— 元件名稱、難度層級、slot 幾何，前端據此顯示元件名與難度、計算血量。
